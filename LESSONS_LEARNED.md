# Lessons Learned
## Day 87 — Cycle 3 HARDEN Week, Day 1: Full Interaction Audit

- **Selector drift is the single biggest tax on Harden-Week harnesses.** My first audit pass against the deployed build had 10 "failures" — every one of them was a wrong selector, not a real bug. Real DOM ids on this codebase: `#blitz-mode-btn` (not `#blitz-btn`), `#create-level-btn` (not `#creator-btn`), `#share-card-modal` (not `#share-modal`), `#cosmetic-modal` (not `#customize-modal`), `#start-daily-btn` (not a generic "Start" text search). Lesson: every Harden Day 1 should start by running a `document.querySelectorAll('button')` walk on the visible screen to enumerate real ids before writing any assertions. Day 73 did this and got it right; Day 87 didn't and burned a pass.
- **`isDaily` is a per-level property, not a GameState property.** I assumed `window.game.isDaily=true` after starting a daily challenge, parallel to `isChallengeMode`, `isLabBench`, `isInfinite`, `speedrunMode`, `blitzMode`. It isn't — daily challenges set `currentLevel.isDaily=true` on the generated puzzle, and the mode flags `isChallengeMode/isLabBench/isInfinite` are GameState properties that read through to the current-level surface. Lesson: when asserting mode flags, check the actual code path — `git grep 'isDaily'` would have shown the property lives on the level, not on the engine. Daily-mode detection should look at `currentLevel.isDaily`.
- **Day 79's dead-identifier list said `InteractiveTutorial.getCurrentStep` was removed, but my initial typeof-check returned `'object'` because of the `(a && a.b)` short-circuit pattern.** When `tut` is a real truthy InteractiveTutorial instance and `tut.getCurrentStep` is undefined, `typeof (tut && tut.getCurrentStep)` evaluates to `typeof undefined === 'undefined'` — correct. The first pass got `'object'` for unclear reasons (possibly a parsing quirk in my expression). Switching to the ternary form `typeof (tut ? tut.getCurrentStep : undefined)` made it explicit and the result returned `'undefined'` as expected. Lesson: when a typeof-check returns a surprising value, replace the short-circuit `&&` with an explicit ternary — it removes ambiguity about what's being passed to `typeof`.
- **HUD cleanup placement asymmetry is real — codify it but don't fix it on a HARDEN day.** Day 61 (Blitz HUD) and Day 74 (Speedrun HUD) both placed their defensive cleanup blocks inside `GameState.showLevelSelect()` (the high-level wrapper). The user-facing back-btn correctly routes through that wrapper, so users never see a stale HUD. But the UI-layer screen transition function `ui.showScreen('level-select')` is BELOW the wrapper and has no cleanup of its own. Any internal caller that bypasses the wrapper (today: none in production code; tomorrow: possibly something Polish/Prune adds) would resurrect the bug. This is a P2 latent observation, NOT a bug — I logged it as LO-1 in BUGS.md for a future Polish day to ship. Lesson: HARDEN week's job is to *find* latent issues like this and *document* them, not to fix them — fixing them risks breaking the very stability we're auditing.
- **`ui.showScreen('level-select')` vs `GameState.showLevelSelect()` is the more general pattern.** Every "showScreen" function has two layers now: the UI-layer transition (rendering, focus, animations) and the GameState wrapper (state-machine bookkeeping like "stop mode timers", "clear HUD", "reset mode flags"). The wrapper is correct; the UI layer is a code-smell asking to be promoted to the source of truth. Lesson: when a future Polish day collapses these two layers, do it transition-function by transition-function (level-select first, gameplay second, etc.), not as a big-bang refactor.
- **The end-of-suite "observations" list is a first-class deliverable.** Day 73's audit found 0 bugs and Day 87 found 0 bugs — but Day 87 still ships value because LO-1 is now documented. Without an explicit "observations" section, a 100% pass rate looks like a no-op day. Lesson: every Harden Day 1 should end with an "observations" section that captures *behavior-correct-but-architecturally-questionable* findings; these accumulate into Polish/Prune week backlogs.
- **The Day 86 CDP harness over `ws@8.20.0` via `NODE_PATH` is now battle-tested across 29 phases.** Originally 9 phases on Day 86; today it scaled to 29 phases / 66 assertions without changing the harness shape. The pattern — `Page.enable` + `Runtime.enable` + `Network.setCacheDisabled` + `Storage.clearDataForOrigin` + `navigator.vibrate=()=>true` + a `pending` map + `evalExpr` helper + `waitFor` helper — is exactly the right shape for any future Harden Day. Lesson: at this point, Day 87's `qa-reports/day-87-qa.cdp.js` is the canonical template for any factory-day CDP audit. Future days should diff against it, not against the Day 82/83/84 harnesses (those are slimmer single-feature scripts).
- **"No code changed" days still produce 6 deliverables.** Day 87 wrote: regenerated `specs/module-health.md`, new `qa-reports/day-87-qa.md`, new `qa-reports/day-87-qa.cdp.js`, updated `BUGS.md` + `LESSONS_LEARNED.md` + `FACTORY_STATE.json`, single commit. Cache-bust and SW version NOT bumped (Day 86 precedent: only bump on real code change). Lesson: the audit-only day's commit footprint is 6 files — it's still a real day's work, just none of it lives in `js/*.js` or `index.html`.

## Day 85 — Cycle 3 Build Week, Day 4: Onboarding Experiment Flag

- **A local feature flag is a class, not a global**: I considered storing variant + counters as a flat localStorage blob and reading/writing it inline at the call site. That would scale poorly the moment a second variant added a different side effect. Building `OnboardingExperiment` as its own class — with `applyFirstLaunch()`, `getVariant()`, `getCounters()`, `reset()`, and an internal normalized state — means future variants are added by inserting one more `if` branch and one more counter field, not by re-spreading variant logic across the codebase. Lesson: even a 150-LOC feature flag earns its own manager class once it touches storage, URL parsing, side effects, and QA inspection. The shape is the same as `WeeklyTournament`, `InfiniteRunManager`, and `NotificationManager` — the pattern is unmistakable by now.
- **Precedence chain belongs in one resolver, not at every read**: My first sketch had each read site check URL → localStorage → default. That meant any future caller could subtly diverge. The clean version: `_resolveVariant()` runs once in the constructor, persists the URL override if present, and from then on every reader just returns `this._state.variant`. The variant becomes a constant for the lifetime of the page, and the precedence is documented in one place. Lesson: any time you find yourself writing the same precedence chain in two callers, the resolution belongs in a constructor or initializer — readers should just read.
- **"No external analytics" means no buffered events either**: I briefly considered queuing counter updates into a `pendingEvents[]` array "in case we ever want to flush them to a backend". That's the start of a slippery slope: queues invite flush helpers, flush helpers invite retry logic, retry logic invites a backend. Counters are simple integers stored in localStorage and never read by anything except the QA panel. Lesson: when a spec says "local counters only", build them as literal local counters — don't pre-build the analytics seam. Adding the queue later when a real backend exists is a 30-minute job; ripping out an accidentally-shipped analytics seam is a multi-day audit.
- **Cold-state restoration via `reset()` beats per-key cleanup**: I exposed `window.__onboardingExperiment.reset()` instead of a constellation of `clearDifficultyKey()`, `clearExperimentKey()`, `clearCounters()` helpers. One named verb ("reset") that restores the cold start in one call — verified by the QA harness with a single call followed by `applyFirstLaunch()`. Lesson: dev/QA surfaces should expose verbs that map to player-meaningful state transitions, not getters/setters for every internal field. Future variants that touch new keys can extend `reset()` from the inside; QA scripts don't need to know about them.
- **`addEventListener` stacks — instrument without monkey-patching**: For `explicit-chooser`, I needed to record which difficulty the user picked inside `ui.showDifficultySelector()`. The naive approach would have been to wrap or monkey-patch `setDifficultyMode`, which would have leaked the wrapper into Settings → Difficulty Mode (a totally separate flow). Instead, I let the existing chooser populate its DOM, then `setTimeout(() => querySelectorAll('#diff-options .diff-option-btn').forEach(...addEventListener(..., {once:true})))` attached a *second* listener on each option — both fire on click, the existing one closes the modal, mine records the pick. Lesson: when you need to observe a UI event in one specific flow, attach a second `addEventListener` rather than wrapping the handler that fires the event. `{once: true}` makes the cleanup automatic.
- **Debug-only Settings surfaces gate via localStorage on `show`, not on init**: I considered checking `signal-circuit-debug` once in `setupSettingsModal()` and rendering the developer section based on the value at that moment. That would break the QA workflow — set the flag, refresh, re-open settings. The fix: re-evaluate the flag on every settings-modal *open*. One extra localStorage read per click is free; the developer experience is dramatically smoother. Lesson: "gated by a flag" surfaces should re-check the flag on every visibility transition, not on first-init. Cold-start chrome is unaffected because the developer section's parent (the Settings modal) is itself gated behind a button.
- **CDP localhost verification has a stable preamble at this point**: Spawn permissive Chromium on port 9301 with `'--remote-allow-origins=*'`, open the page via `PUT /json/new?<url>`, run a tiny `Runtime.evaluate` harness with `suppress_origin=True`, pre-install `navigator.vibrate = () => true` to silence headless gesture-policy noise, and clear localStorage between phases by navigating with `localStorage.clear()` inline. Day 72/76/78/82/83/84 used the same shape. I should consider promoting this into a reusable harness module (`tools/cdp-harness.py`) for Cycle 4 so each day's QA script starts from a 30-line skeleton instead of a 250-line ad-hoc rewrite. Lesson: every Cycle 3 day that re-implements the localhost CDP preamble is paying down the same tax. Time to amortize it.

## Day 84 — Cycle 3 Build Week, Day 3: Lab Bench II Seed Pack

- **Layer constraints on the existing state machine, not on a new mode**: Day 70's Lab Bench machinery (attempts, exhausted, firstTryLocked, Reset Lab, RUN button relabel, Quick Test suppression) already covered "design first, submit once". Adding gate-cap and required-gate constraints is just one more check inside `runSimulation`/`runQuickTest` after `_consumeLabAttempt()`, plus one extra HUD chip. No new flow, no new modal, no second tutorial. Lesson: when a designer asks for "a new puzzle constraint", the cleanest implementation usually layers a single validator on top of the existing submit gate — don't reach for a sibling state machine.
- **Per-level optional fields beat global subtype flags**: I chose to put `labConstraint`, `gateHardCap`, and `mustIncludeGate` directly on the level objects (all optional). Each new constraint is one field; legacy lab levels 36–40 ignore them transparently because `updateLabHud()` reads with `if (level.labConstraint)` and the validator returns ok when the fields are absent. Lesson: opt-in level fields scale better than a `subtype: 'nand-only'` enum — the enum forces the validator to know every variant, but field-checks degrade gracefully when a level omits a constraint.
- **Submission failures should still cost an attempt in lab mode**: I considered short-circuiting `_consumeLabAttempt()` when the constraint validator fails, so a player could fix and re-submit "for free". That actively undermines the "submit once" philosophy: students stop reading the constraint chip and just spam RUN. Charging the attempt makes the chip matter. Lesson: when a feature enforces a discipline, every cheap escape hatch quietly removes the discipline — keep the cost honest.
- **Constraint copy belongs in the HUD chip, not the description blurb**: The level `description` is read once and skimmed. The HUD chip lives next to the gate count and the submission counter, so the player sees the constraint every time they look at the budget. Lesson: persistent UI is the right home for rules the player must keep in working memory; descriptive copy is the right home for the story/setup.
- **Headless QA harnesses need to know about ID-based wiring**: My first build attempts at `g.wireManager.add(A, n1.inputs[0])` threw `TypeError: Cannot read properties of undefined (reading '0')` because Wire is constructed from `(fromGateId, fromPinIndex, toGateId, toPinIndex)` — not pin objects. The clean fix: a `window.__addWire(fromId, fromPin, toId, toPin)` helper injected at QA start that uses the real `Wire` constructor and respects "one wire per input pin". Lesson: when synthetic gameplay is part of the QA story, add a thin, named wiring helper to the harness rather than poking at private data structures — future days will reuse it and won't re-discover the ID shape every time.
- **`offsetParent !== null` is the wrong visibility test in some headless contexts**: My first L41 chip visibility check used `el.offsetParent !== null` and reported false even though the chip was clearly visible to the validator. Switched to `el.style.display === ''` (empty string == default, matches my toggle) and it passed. Lesson: in headless CDP runs without a real viewport, `offsetParent` can return null for layout-correct elements; check `style.display` or `getComputedStyle().display` for inline-style toggles.
- **Stale localStorage from prior QA runs silently breaks `seedProgress`**: My second QA pass's `seedProgress(43)` returned `seeded: 0` because the previous run left the user-data-dir populated with stars=3 for every level; the monotonic guard correctly skipped them. The fix: wipe `localStorage` then trigger a fresh reload before the test sequence. Lesson: every CDP QA harness that persists across runs needs a `localStorage.clear()` + reload preamble — not just a cache-bust query string, because storage outlives query params.

## Day 82 — Cycle 3 Build Week, Day 1: Shareable Circuit Snapshot Cards
- **Social hooks are strongest when they reuse earned artifacts**: The game already had a generic share-card generator (Day 31) and real solved-circuit preview data (Day 43). Combining them produced a social feature without adding a new top-level button, new storage schema, or external service. Lesson: before building a new sharing surface, look for an existing artifact the player has earned and make that artifact visible.
- **Completion-screen actions preserve prune wins**: I did not add a level-select social button. The `📸 Share Card` action remains gated behind solving a campaign level, which keeps the Day 78/80 cold-start surface clean. Pattern: viral/social features should appear at peak intent (after success), not on first launch.
- **Canvas share cards need graceful capability tiers**: Browser support differs across Save, ClipboardItem image copy, and Web Share file sharing. The implementation now uses a tiered strategy: file share when `navigator.canShare({files})` allows it, text/link share otherwise, ClipboardItem image copy when available, and text fallback instead of throwing. Lesson: share features should degrade by capability, not by user-visible failure.
- **Headless haptics can pollute console-error counts**: Programmatic Quick Test solves trigger `navigator.vibrate()` without a user gesture, producing browser policy errors even though gameplay is correct. The Day 82 harness stubs `navigator.vibrate` so console checks measure JS health rather than headless gesture policy. Lesson: when a feature is verified through synthetic actions, stub browser APIs whose failure is explicitly gesture-policy-related and not product logic.
- **Pixel diversity is a cheap canvas assertion**: Rather than eyeballing screenshots, the CDP harness sampled the right-side share-card panel and counted non-dark/tealish pixels. That proves the snapshot panel drew real content while staying independent of exact coordinates. Pattern: for generated canvas art, assert dimensions + coarse pixel characteristics; exact image matching is overkill and brittle.

## Day 80 — Cycle 2 Prune Week, Day 4: Polish Sprint
- **Diff-based reveal pulse beats event-based hooks**: I considered firing the staircase pulse from `completeLevel()` (the moment a player finishes the 6th/9th/12th/15th/18th level), but that ties the animation to one specific code path. Instead, I added a `_lastGatingState` snapshot inside `applyProgressGating()` itself — any caller that re-runs gating gets a free diff. Works for: cold-start gating (suppressed because `prev` is undefined), `completeLevel` (which already calls `applyProgressGating`), `seedProgress` (Harden helper, free coverage), and any future caller. Lesson: when an animation should fire "whenever X transitions Y→Z", diff the function that establishes X instead of the events that drive X.
- **Cold-start suppression is a one-line check, not a state machine**: My first sketch had a `_firstGatingDone` boolean, an init flag, and a special-case for `seedProgress`. The actual implementation: `if (prev && visible && prev[id] === false)`. Three operands. Lesson: suppression of "first time" effects is usually `if (prev)` — don't reach for booleans when the prior state itself is the signal.
- **`overflowPopIn` is the same shape as every prior popover animation — reuse the keyframe pattern**: 180ms ease-out, opacity 0→1 + transform translateY(-4px)→0 + scale(0.98)→1. This is the same shape as `screen-zoom-in` (Day-#92), the welcome-toast slide, and the achievement-toast slide. Lesson: each polish day should have one "popover open" keyframe and reuse it. We don't have a `--anim-pop-in` custom property yet, but if Day 81+ adds another pop-up, that's the right moment to extract one.
- **`:focus-visible` is the cheapest a11y win, but it only paints on real `<button>` elements**: My `.level-btn:focus-visible` rule was harmless but never fires because `.level-btn` is a `<div>` with `tabindex=-1`. The rings that actually paint are on `.level-overflow-btn`, popover menu items, `#open-settings-btn`, `#run-btn`, `#back-btn`, `#how-to-play-btn`, `.tool-gate` — all real buttons. Lesson: when adding focus styles, sanity-check the underlying element type. A future polish day can add `tabindex=0 role="button"` to `.level-btn` so the ring becomes useful for keyboard players.
- **`@media (prefers-reduced-motion)` checks need to query the nested rule, not the cssText**: My initial verification did `cssText.includes('animation: none !important')` and got `false`, panicking that the welcome-toast reduced-motion rule wasn't there. The rule exists; `CSSMediaRule.cssText` only includes the media wrapper text, not the nested rules. The right query is `for (const sub of rule.cssRules) ...`. Lesson: stylesheet introspection through `cssText` flattens at the top level but not inside `@media`/`@supports`/`@layer`. Use `rule.cssRules` to walk into wrappers.
- **CDP-driven SW eviction beats waiting for hash mismatch**: The first run hit the stale-cache trap (the SW was serving last-deployed assets from `signal-circuit-v54` over the new `v55`). Reloading didn't help — the SW intercepts and returns its cache. Workaround: run `getRegistrations()/unregister()` + `caches.keys()/delete()` from JS, then open a fresh tab with a cache-busting query string. The new tab bypasses the dead SW long enough for the page to register `v55` and take over. Lesson: any local-server CDP verification day where the URL is the same as the live PWA must explicitly evict the SW before relying on "reload to see new code". Add this to the verification preamble for future Polish/Harden days.
- **Polish budget is ~+70 LOC, not strictly net-neutral**: PRUNE Week's rule of thumb is net-negative LOC, but Day 4 (Polish) is the explicit exception in the prompt ("polish often adds a few lines for transitions/animations; single commit"). My final delta is +66 (ui.js +12, audio.js +5, style.css +49). All additive (keyframes + diff state + focus rings + mobile @media block + audit comment). Lesson: PRUNE Week's net-negative covenant is a *weekly* total — with Day 79's −180, this week is still way ahead.

## Day 79 — Cycle 2 Prune Week, Day 3: Code Cleanup
- **Three-pass dead-code detection beats reading focus notes alone**: Day 79's nextCycle.focus listed 3 expected targets (showFirstLaunchDifficultyModal, weekly-puzzle-btn, view-solution-btn shared CSS). A 30-line Python script that walked every class method in every JS file and grep-counted usage across the project surfaced 4 *additional* orphan methods I would have missed (`checkLightning`, `checkEclipseRun`, `checkArchitect`, `isMythic`, `_showHud`, `getCurrentStep`). Lesson: focus notes are a starting list, not the full list — always pair them with an automated orphan scan. The script is ~30 lines, runs in <1s, and finds the bones that human review skips.
- **"Hooks added Day N" claims need a usage check, not a trust check**: Day 71's status said `checkLightning/checkEclipseRun/checkArchitect` were wired into the completion paths. The actual wiring inlined `this.unlock('mythic_X')` directly at the trigger sites and left the wrapper methods dangling. Both work — the wrappers are just dead duplicates. Lesson: when a previous day claims "X is hooked into Y", verify the wiring goes through the *named API* you'd expect, not through an inline duplicate. If both exist, the wrapper is almost always the orphan.
- **Orphan CSS rules cluster around renamed-but-not-deleted DOM IDs**: `#confirm-modal-box`, `#creator-content`, `#mastery-content` all once existed; refactors renamed them to `#confirm-modal-content` / `#creator-config-content` / `#mastery-tree-content` but left the CSS selectors pointing at the dead IDs. The selectors were compound (`body.light-mode #foo, body.light-mode #bar, ...`), so the rule still applied to the surviving IDs in the list — nothing visibly broke. Lesson: when refactoring an element's ID, grep CSS for *both* the old and the new name; orphans hide inside compound selectors because the rest of the comma list still works.
- **`#mute-btn` and `#gate-count-display` were never-rendered survivors of design pivots**: Neither ID has ever appeared in `index.html` (going back through git), but CSS rules existed for them — likely written when an early design called for a sandbox gate counter and a manual mute toggle that were eventually replaced by the settings modal and the SFX/Music sliders. Lesson: "unused CSS" can mean three different things — (a) used to exist and was removed, (b) was speculatively styled and never built, (c) is dynamically constructed via `class='trace-val-' + value`. The orphan scan needs to distinguish (a/b) from (c); for (c), search for the prefix as a string literal in JS.
- **A code-cleanup commit should land as a single commit, not the 5-cut atomic style**: Day 65 set this precedent (single commit, 14 JS + 10 CSS removals); Day 78 used 6 atomic commits because it was 5 distinct *user-facing* cuts. Day 79's removals are interlocking (deleting the button DOM, the click handler, the gating call, the CSS class, the notification path — all part of one logical "retire the dead feature surface") and don't map cleanly to user-visible slices. Lesson: atomic commits are for *user-observable* changes, not for *cleanup* changes. A single "Day N: code cleanup — remove X JS + Y CSS" commit is cleaner to revert and easier to scan in git log.
- **Net-negative LOC target is easier than it sounds when you give yourself permission to remove notification scaffolding**: I almost stopped after the 3 listed targets (which would have been ~80 LOC net negative). Walking *further* into the weekly-notification scaffolding (the toast, the week-number helper, the settings entry, the localStorage key) was what pushed the net to −180. Lesson: when a feature is retired, the cleanup blast radius is usually 3–5 collaborators deep — the button, the click handler, the gate, the storage key, the toast, the settings toggle, the helper. Walk every single one before declaring the cleanup done.
- **Verification matrix should include both "X is gone" and "X's replacement still works"**: 12 "is it gone?" checks + 3 "does the replacement still work?" checks. The 3-check regression sweep caught nothing but provides the floor: Settings→Difficulty still opens the chooser (Day 56), Tournament still loads (Day 72), Notifications still have Daily + Streak entries. If any of those broke during the cleanup, the regression sweep would catch it before commit. Lesson: every cleanup day's CDP suite needs an "is the replacement intact?" tail — not just "is the dead code gone?".

## Day 78 — Cycle 2 Prune Week, Day 2: Execute Tier 1 Cuts
- **Overflow `⋯` menus need anchor toggling AND outside-click dismissal in one pattern**: The new per-card overflow popover (`_showLevelOverflowMenu` / `_hideLevelOverflowMenu`) opens against an anchor `<button>` and registers a `document.addEventListener('click')` listener on the **next tick** to avoid the opening click self-dismissing it. Toggle semantics live in the anchor's own handler (re-clicking the same anchor calls `_hide`, clicking a different anchor closes the old popover and opens a new one against the new anchor). Lesson: any "single popover lives at a time" UI needs three pieces — a manager-owned reference (`this._levelOverflowAnchor` / `this._levelOverflowPopover`), a one-tick-deferred outside-click listener, and an anchor-aware toggle path. Skipping any one introduces a glitch (instant dismissal, two popovers open, or sticky popover after click).
- **Per-card adornments belong inside the card's positioning context, not the document body**: I appended the popover to the card via `card.appendChild(pop)` and set `card.style.position = 'relative'` if needed, so the popover scrolls with the card. Appending to `document.body` and absolutely positioning by computed-rect works but breaks under window scroll and the level-select's staggered fade-in. Lesson: when a popover "belongs" to one element visually, parent it to that element — you get scroll/transform tracking for free.
- **Staggered gating is just a sorted threshold list, not a tier rewrite**: `applyProgressGating()` already had a tier1/tier2/tier3 trichotomy. Going from 2 thresholds to 5 (g6/g9/g12/g15/g18) added 3 extra constants, 7 extra `setVis` rebindings, and zero structural change. The function shape stays "flat list of `setVis(id, gate)` lines". Lesson: when a UI ramp feels too steep, the fix is rarely a redesign — it's usually adding 1-2 more threshold values and re-bucketing the existing ID set into them. Document the new bucket layout in a code comment so future audits can re-tune without re-reading every `setVis` call.
- **Retiring a feature beats migrating it when a strict superset already exists**: Tournament (Day 72) was always going to subsume Puzzle of the Week. Cycle 1 left PotW alive at Tier 2 "for now"; Day 78 just removed the button surface (`setVis('weekly-puzzle-btn', false)`) without touching the underlying `generateWeeklyPuzzle()` helper (Tournament still calls it) and without migrating saved PotW bests. The leftover localStorage data is inert. Lesson: if feature A subsumes feature B and B has only a small amount of user data, retire B by hiding the entry point and leaving the data — don't migrate, don't delete. Migration is the cost; subsumption is the win.
- **Cold-start ceremonies should default to the recommended choice + a toast, never a modal**: Cut #5 replaced the first-launch difficulty chooser modal with `setDifficultyMode('standard')` plus a 4.5s toast pointing at Settings. Modal → toast is a one-way ratchet during Prune weeks — any modal that gates "choose your X" between cold start and the first puzzle is presumptively a tax. The toast preserves discoverability for the small fraction of players who want a non-default. Lesson: if a modal exists to suggest a default, *do* the default and surface the *change path*, don't ask the user to confirm the default.
- **Suggestion-storage flags must double as default-storage flags**: I considered adding a separate `signal-circuit-difficulty-suggested` flag (so silent-default and explicit-choice could be distinguished), but realized `DIFFICULTY_KEY = 'signal-circuit-difficulty-mode'` already serves both roles: presence means "don't auto-prompt again", and the value is the actual mode. Lesson: don't invent parallel flags when an existing storage key already gates the prompt path. The fewer keys, the fewer migration surfaces.
- **Cold-state hint copy should key off `level.id`, not just `hintsUsed`**: The original `if (gs.hintsUsed === 0)` branch fired the meta-rule footer on every fresh load of every level, including L1 step 1 where the player has no concept of stars yet. Adding `level.id < 4` as an early-return inside that branch keeps the data path simple (no new state, no localStorage flag) while gating the surface by content-context. Lesson: when a piece of UI copy makes sense "after the player has learned X", and X is taught at a specific level, just gate the copy by `level.id` rather than tracking an event-based flag. The level number is the canonical "how far in" signal.
- **`git restore` wipes ALL pending changes, including the ones you wanted to keep for cache-bust**: I saved the working state to `/tmp/sc-d78-final/`, `git restore`d to start clean for atomic commits, replayed each cut in order, and at the end realized the cache-bust bumps had been wiped too. The 11 `?v=` references in `index.html` and `CACHE_NAME` in `sw.js` had to be re-applied as part of the final commit. Lesson: for multi-cut atomic-commit workflows where some changes are "shared across all cuts" (like cache busts), either commit the cache bump FIRST as a trailing-cut precondition, or save it separately and re-apply. Don't trust that `git restore` will leave the cache bumps alone — it won't.
- **CDP verification on localhost still needs the policy-block workaround**: Day 72 / Day 75 / Day 76 all logged the same pattern: `browser action=open http://localhost:8901/` returns `browser navigation blocked by policy`, but `curl -X PUT http://127.0.0.1:18800/json/new?<url>` opens a CDP tab directly. The fallback works identically through Cycle 2. Lesson: factory localhost verification is a 4th-class browser path that we should not keep relearning — codify the workaround in TOOLS.md or the next skill, not just LESSONS_LEARNED.

## Day 76 — Cycle 2 Harden Week, Day 4: Fix Everything (Closing the Day 75 learnings)
- **A bug-free Harden Day is a chance to close verification-only learnings, not a free day**: Day 76 inherited an empty Open Bugs queue. The orchestrator prompt's "open in browser" step does not apply when the browser tool refuses localhost. Use raw CDP from a permissive Chromium spawned via shell instead.
- **Permissive headless Chromium for localhost verification**: `/Users/openclaw/Applications/Chromium.app/Contents/MacOS/Chromium --headless=new --remote-debugging-port=9301 '--remote-allow-origins=*' --user-data-dir=/tmp/sc-day81-chrome` is the established harness. The `--remote-allow-origins=*` flag (single-quoted in zsh!) is required — the OpenClaw-managed browser at port 18800 rejects CDP WebSocket connections from any origin including its own. Use a separate port (9301) so you don't fight the managed instance.
- **zsh globbing eats `*`** in unquoted CLI flags — `--remote-allow-origins=*` fails with `no matches found`. Always single-quote: `'--remote-allow-origins=*'`.
- **GameState IS `window.game`** (not `game.gameState`). The constructor name is `GameState` but it's exposed directly. Prior Day notes referenced `game.gameState.seedProgress()` in narrative text but actual calls are `game.seedProgress(count, opts)`. The Day 76 dev helper lives on `GameState.prototype.seedProgress` directly.
- **Screen IDs are `*-screen` suffixed**: `#level-select-screen`, `#gameplay-screen`. Not `#level-select`. There's a `#level-select-content` element inside the screen. Several `[id*=level-select]` matches exist; always disambiguate.
- **Screens use inline `display` styling** rather than a `.active`/`.visible` class. Query via `getComputedStyle(el).display !== 'none'` to determine which screen is active. There are NO elements with class `.screen` — that's a different convention.
- **JSON files with surrogate pairs need `ensure_ascii=True`** on Python 3.14. The existing FACTORY_STATE.json mixes `\uXXXX` escape sequences for emojis (which are surrogate pairs like `\uD83D\uDCD0` for 📐). Reading those is fine (`json.load` decodes them), but writing them back with `ensure_ascii=False` produces lone surrogates that `f.write` rejects with `UnicodeEncodeError: surrogates not allowed`. Use `ensure_ascii=True` (default) to preserve the round-trip, OR use full Python string literals like `'\U0001f4d0'` (32-bit) instead of surrogate pairs.
- **`edit` tool partial failure leaves the file corrupted**: When `edit` reports "Could not find edits[N]", earlier edits[0..N-1] in the same call may have already mutated the file. Always verify `python3 -m json.tool` validity after a multi-edit call to JSON; if invalid, `git checkout -- FILE` and reapply via a Python script that round-trips through `json.load`/`json.dump` for safety.
- **`_lastGatingState` undefined on cold start is by design**, per Day 80's pulse-suppression branch. The staircase pulse functionally works — verified across 5 seed transitions — but the variable doesn't appear on `game` until `applyProgressGating()` runs at least once after a seed. If a future Polish day wants `game.dev.lastGating` for debugging, add an explicit assignment.

## Day 83 — Cycle 3 Build Week, Day 2 (Tournament Backend Adapter Shell)
- **Adapter at the call site, not inside the class.** The cheapest place to add a transport boundary is at the *one or two* lines where gameplay actually invokes `submitScore`. Wrapping `WeeklyTournament` itself would have meant touching ISO-week math, leaderboard generation, archive logic, and persistence. Wrapping the *call site* is a 1-line change per site plus an adapter file. Surgical.
- **`describe()` is the cheapest UX seam between local and cloud.** A single text method on each adapter (`describe()` returning a short label) means the UI never has to branch on `getMode()`. The tournament screen just renders whatever the adapter says it is. Future modes (live, degraded, offline) can ship without changing UI code.
- **Worker-shaped doesn't mean Worker-wired.** The `RemoteTournamentAdapter` exists to *prove the seam*, not to talk to a real backend. Its `submitScore` forwards unconditionally to `LocalTournamentAdapter` today. Doing it any other way risks accidental network writes during development or QA. Only flip to live mode when auth, anti-cheat, rate limits, and schema versioning are explicitly designed in a separate day.
- **Sync-shaped return preserved for now.** Both completion call sites read `submission.score` / `submission.rank` synchronously and immediately update DOM and achievements. Widening to a `Promise` was tempting but would have rippled through `updateResultDisplay`, `updateStatusBar`, achievement toasts, and the celebration system. That refactor belongs to the day when a real `fetch()` lands, not to the seam-introduction day.
- **Detection order: window override → localStorage → default.** Three layers, each with a clear use case: `window.__SC_TOURNAMENT_BACKEND__` for ad-hoc QA in the dev console (no reload required after re-init), `localStorage('signal-circuit-tournament-backend')` for stable testing across reloads, default `'local'` for production. Validating both `'remote'` and `'local'` as legal localStorage values keeps the surface tight and prevents typos from silently switching modes.
- **Expose adapter classes on `window` for QA.** `window.TournamentBackend`, `window.LocalTournamentAdapter`, `window.RemoteTournamentAdapter`, `window.selectTournamentBackend` cost nothing and let the QA harness re-instantiate the adapter without reloading the page. This is invaluable for testing the toggle path — a full reload would re-create `GameState` and lose the test state.
- **Defensive fallback at the call site.** `(this.tournamentBackend || this.weeklyTournament).submitScore(…)` survives the edge case where some hot-patched build instantiates a `WeeklyTournament` but no `tournamentBackend` (e.g., during incremental rollouts). The fallback is a single `||`; the cost of paranoia is zero.
- **Tournament data slate reset for deterministic QA.** Achievement assertions like `tournament_crowned` are sticky in localStorage. The QA harness explicitly clears `signal-circuit-tournament-v1` and re-initializes `weeklyTournament.data` before each submission test so rank/percentile/podium logic gets a fresh slate per run.
- **`isLive()` and `getMode()` are not the same thing.** `getMode()` is the *intent* ("local" / "remote-ready" / "remote-live"); `isLive()` is the *actual current capability* (true only when a real network round-trip is wired and reachable). A remote-mode adapter with a configured Worker URL but no live mode wired still returns `isLive() = false`. This split lets a future Day flip `isLive()` to true without renaming modes.
- **Calm UI label > marketing copy.** A single soft-grey 11px line under the week label communicates the mode without yelling. The label changes the *information density* of the tournament screen by exactly one row, which is the right cost for the right amount of disclosure.
- **Day 82 `navigator.vibrate` stub pattern is reusable.** Every CDP QA run that exercises gameplay completion should pre-install `navigator.vibrate = () => true` to silence the browser's user-gesture warning. Day 82 and Day 83 both used it; expect every future completion-touching QA to need it.

## Day 86 — Cycle 3 Build Week, Day 5 (Module Split Foundation)
- **Measure before you migrate.** ES-module conversion of ten inter-referencing global-script files in one day was the wrong shape for a BUILD-week timebox. Shipping the *measurement* layer first (a regex-based module-health report) costs ~240 LOC of pure Node, produces a baseline that every future day can re-run against, and exposes the actual coupling hotspots before any extraction is attempted. The acceptance criterion explicitly allowed "report OR reduction" — picking both halves of an "OR" when the cheap half unlocks the expensive half is the right move.
- **The cheapest demonstrable reduction is "delete a dead global".** `const WIRE_COLORS = WIRE_COLORS_DEFAULT;` had fan-in=0 (referenced nowhere outside its own definition) AND was unused within `wires.js` itself. That's not "internal state" — that's dead code. Removal is provably side-effect-free. One line, one global gone (111→110), report-driven, and the diff is a one-sentence story.
- **Regex-based parsing is fine for coupling trends.** AST parsing would catch identifier mentions inside string literals and comments and exclude them. Whole-word regex doesn't. That over-counts coupling on the margin and never under-counts — which is the *correct* direction for a trend metric. Don't pull in a parser dep when you're measuring "is this number going up or down over time."
- **`gates.js` is the natural first ES-module extraction.** Fan-in=4 (referenced by 8 of 10 files), fan-out=0 (depends on nothing else). It's the dependency graph's purest leaf. When the time comes to do real ESM, start there — its single-file extraction can't break anything upstream because it has no upstream.
- **`ui.js` is the natural last extraction.** Fan-out=25 across 5 files, fan-in=1 (just `UI`). It consumes everything else. Migrating it first would create an island of ES-module code that imports from nine still-script-tag files — exactly the kind of dual-world mess that makes incremental migrations stall. Save it for after every other module is converted.
- **`main.js` is the orchestrator and the biggest lever.** 60 globals defined in one file (54% of the 110-global total), fan-in=6 (other files only reach in for a small surface), fan-out=24 (it reaches everywhere). The actionable insight: most of those 60 globals are probably *internal* to main.js but happen to live at module-top by accident. Wrapping main.js in an IIFE — or moving its locals inside its `GameState` class — would dwarf any other coupling reduction we could ship. That's a future Polish/Prune day with its own QA budget.
- **CDP harness over `ws@8.20.0` via `NODE_PATH` is the right shape for localhost QA.** `node_modules` isn't checked in to this repo and we won't add it just for QA. Pointing `NODE_PATH=/Users/openclaw/src/openclaw/node_modules` at OpenClaw's existing `ws` install gives the harness everything it needs without any dependency surface in the project itself. Same pattern works for `puppeteer-core` if a future day wants higher-level helpers.
- **`game.ui` ≠ `window.ui`**, **`game.inputNodes`/`outputNodes` ≠ `game.gates`**, **`game.wireManager` ≠ `game.wires`**. These were each a stumble during the Day 86 harness build — each lesson in the form "the obvious-looking property name isn't the real one." The QA harness now uses the real names: `game.ui.showScreen`, `game.inputNodes`/`outputNodes` (IONodes live in dedicated arrays, not in `gates`), `game.wireManager.wires`, `game.addGate`/`addWireFromData`. Pin these for future harness writers.
- **`runSimulation()` is async and animation-heavy; `simulation.runAll()` is the QA shortcut.** The synchronous helper at `js/main.js:3872` returns `[{pass: bool, …}]` for every truth-table row, no audio, no tension animation, no UI side-effects beyond `updateTruthTable`. Use it for circuit-correctness assertions; reserve `runSimulation` / `runQuickTest` for end-to-end celebration-path checks.

## Day 88 — Cycle 3 HARDEN Week, Day 2: Level Playthrough

- **`startLevel(N)` vs `loadLevel(N)` matter for harness work.** `loadLevel(N)` updates `currentLevel`, repositions IO nodes, redraws the canvas — but does NOT call `ui.showScreen('gameplay')`. From the level-select screen, calling `loadLevel(1)` alone leaves `screen-active` on `level-select-screen` and any DOM probe of `#gameplay-screen` returns `display:none`. To actually transition, the QA harness must call `startLevel(1)` (which calls `ui.showScreen('gameplay')` first, then `loadLevel(1)` internally). Lab Bench level loads (L41/L42/L43) need the same treatment, otherwise the `#lab-hud` chip checks read off a hidden HUD.
- **`game.difficultyMode` is the property; there is no `getDifficultyMode()`.** Day 56 added the difficulty state machine with `setDifficultyMode(mode)` + `isRelaxedMode() / isHardcoreMode()` helpers but never exposed a getter. To read the current mode in a QA harness, just touch `window.game.difficultyMode` directly. Mirror lesson: don't trust auto-complete; check the actual class definition.
- **The active screen class is `.screen-active`, not `.active`.** The CSS pattern in `ui.js:771-810` adds/removes `screen-active` on `<div class="screen">` elements. A harness querying `document.querySelector('.screen.active')` silently returns `null` everywhere and every screen-check assertion fails with a misleading `screen=undefined`. The right selector is `document.querySelector('.screen-active')` (single class, hyphenated). Pin this — it bit Day 88's first harness draft on every screen-state assertion.
- **DOM id drift you have to memorize: `#start-daily-btn` (not `#daily-start-btn`), `#speedrun-btn` (not `#speedrun-mode-btn`).** Day 87's harness was right because it was modeled on Day 73's audit which had the same ids. Day 88's draft slipped on both — symmetric mistake where the modifier word ended up in front of the noun instead of behind it. The Daily, Random, Blitz, and Sandbox buttons all follow the `<thing>-btn` pattern, but Speedrun is just `#speedrun-btn` (no `-mode-`), and Daily's *start* button drops the `-mode-` too. A future grep-the-DOM-id-list-before-typing habit would save a debugging round.
- **Headless chrome-for-testing profile persists across runs — wipe localStorage at harness start.** The `--user-data-dir=/tmp/sc-cdp-profile` directory survives between invocations. That's fine for the chrome process itself, but the *page's* localStorage also persists, which means hint-token state and progress from a previous QA run leak forward and can deplete a token budget mid-test. Day 88's Phase 4 (hint button) was 100/100 on the first run and flaked to 99/100 on the second (`hintsUsed` stayed at 0 because `spendHintToken()` returned false). The fix: at the top of the harness, after the first navigation, clear all `signal*` localStorage keys + `communityPlays` + `blitzMode` + `speedrunBest`, then re-navigate. This codifies the chrome profile is shared but the *page state* is per-run.
- **Re-deriving truth tables from semantics > trusting the static table.** For HARDEN audits, the highest-signal check on `levels.js` is to encode the level's *stated* logic (e.g. "Majority of 3", "2:1 MUX with selector S=0→A else B", "3-input XOR / odd parity", "2-bit ripple adder") in the QA harness as pure JS functions, then call `combos(n).map(fn)` to generate the expected table, then row-by-row compare to the level's actual `truthTable`. This catches the case where someone edits a level description but forgets to update the table (or vice-versa) — a class of regression that *truth-table-matches-itself* would miss entirely. 12 levels × 6 checks = 72 assertions of free signal that took 80 LOC of harness to set up. Worth it.
- **HARDEN Tuesday verdict template: 13 phases / 100 assertions / 0 console errors / 0 user-facing bugs.** Day 88 mirrors Day 87's no-code-changed shape — cache-bust stays pinned, sw version stays pinned, the deliverable is a QA report + harness + state update + Telegram summary. Treat HARDEN Tuesdays as "play through everything, document everything, change nothing." If a bug surfaces, Thursday is the fix day; Tuesday's deliverable is the evidence trail.
