# Lessons Learned

## Day 120 — Cycle 5 PRUNE Week, Day 3: Code Cleanup

- **The two-day split pays off: removing an orphan is trivial when the prior day created it deliberately.** Day 119 Cut #1 removed the user-facing `My Best` surface and *named the orphan it left behind* (`_renderTournamentMyBest()` → "deferred to Day 120"). Today's removal was a 30-second find because the breadcrumb already existed. Lesson: when a cut orphans a helper, write the orphan's name into the commit/state so the next day's sweep is a lookup, not a re-discovery.
- **Before deleting a CSS rule, prove the *removed function* was its sole consumer — don't grep the class name alone.** Four tournament classes were only used inside the orphaned renderer, but three sibling classes (`.tournament-badge`, `.tcard-title`, `.tournament-section-title`) share the same prefix and are still live (archive pill, puzzle title, index.html headers). A naive `grep tournament-` would have over-deleted. The safe method: grep each candidate in `js/` + `index.html` *excluding the function being removed*; if zero hits remain, it's dead. The harness then re-confirms via `document.styleSheets` rule-scan (P3 asserts removed=0-rules AND kept=still-present).
- **A stylesheet rule-scan is a better dead-CSS check than a source grep.** P3 enumerates every `selectorText` across all loaded sheets and asserts the removed selectors match 0 rules while the kept ones survive. This catches the real runtime state (what the browser actually parsed) rather than what a text search thinks is there — e.g. it would catch a rule accidentally surviving inside a `@media` block a line-grep missed.
- **Splitting a compound light-mode selector is the fiddly part of CSS pruning.** `.tournament-best-card` was dead but shared a rule with the live `#tournament-puzzle-card` (`body.light-mode #tournament-puzzle-card, body.light-mode .tournament-best-card {...}`). You can't delete the block — you have to surgically drop one selector from the comma list. Same care as removing one identifier from a destructuring pattern.
- **Code Cleanup day is the right place to scaffold the *next* feature's map without shipping it.** Day 121's collection-modal merge needs a verified DOM/handler map; gathering it during the low-risk cleanup day (when I'm already reading ui.js end-to-end) means Day 121 opens with the inventory done. Planning artifacts (`specs/day-121-*.md`) are net-zero risk and front-load the expensive discovery.

## Day 119 — Cycle 5 PRUNE Week, Day 2: Design Simplification

- **A typed-confirm gate must make the disarmed OK click a true no-op, not just a styling change.** The whole point of Cut #2 is that a mis-tap can't wipe the save. It's not enough to grey out the button — the click handler itself must re-check `matches()` and return early, AND the button must carry the real `disabled` attribute. The harness proved both layers (P3.e disarmed-on-wrong-word + P3.f disarmed-click-leaves-progress-18→18). Defense in depth: visual (`.is-disabled`) + attribute (`disabled`) + handler guard (`if (!matches()) return;`).
- **Reuse the existing modal DOM for a new confirm variant; don't fork a second modal.** `showTypedConfirmModal` injects a default-hidden `#confirm-modal-input` into the *same* `#confirm-modal` the plain `showConfirmModal` uses, and `cleanup()` re-hides it. This keeps the destructive-confirm and ordinary-confirm flows visually identical and means the plain path is provably unaffected (P3.k: input stays `display:none` for `showConfirmModal`). One modal, two entry points, beats two modals.
- **"Hide, don't dim" is the right call for zero-state tabs — but add a strand guard when you do.** Cut #3 replaced the Day 104/111 `.empty` dim with `display:none`. The risk a dim doesn't have: a hidden tab could still be the *active* pane if some code path routes to it. The fix is a one-line guard in `_switchStatsTab` (route `cards`/`tournament` → `overview` when their count is 0). Hiding a surface always needs a paired "and never select the hidden surface" guard.
- **De-duplication cuts leave orphans — split the removal across two days deliberately.** Cut #1 removed the `My Best` tab + pane + its dispatch line, which orphans `_renderTournamentMyBest()`. Rather than rip everything out in one commit, the Design-Simplification day removes the *user-facing surface* and the Code-Cleanup day (Day 120) removes the now-dead *helper*. This keeps each commit a single clean intent (IA change vs. dead-code sweep) and matches the PRUNE-week day rotation.
- **Design-Simplification day lands net-positive when the cut is a safety feature — and that's fine.** PRUNE week's mandate is net-negative code, but Cut #2 (typed-confirm) is genuinely additive (~88 functional lines: method + input + CSS). Day 103 set the precedent ("+50 net, comment-stripped ≈ net-zero"). The two surface-removing cuts (#1, #3) offset some of it; the safety win justifies the rest. Don't contort a real safety feature to hit a line-count target.

## Day 109 — Cycle 5 BUILD Week, Day 3: Lab Bench III mini-chapter (L46-L50) with fan-out budget

1. **Fan-out budget is the right "third constraint axis" because it forces signal-sharing pedagogy.** The first two axes (palette restriction via `availableGates`, gate hard cap via `gateHardCap`) shape what you place. Fan-out shapes how you wire — it punishes duplicate-source-to-many-destinations and rewards intermediate buffers. The naive 3-output identity-echo (L46) is impossible with `maxFanOut: 2` until you introduce a 2-stage NOT buffer; that’s the entire teaching moment. **Lesson:** when picking a new constraint axis for Lab Bench N, look for the one that constrains a different dimension of the design space than existing axes. Don’t add a second "how many gates" constraint — add a constraint that bends "how you wire" or "which inputs/outputs you can use".

2. **Fan-out keying by `(fromGateId, fromPinIndex)` is forward-compatible with multi-output gates even though no lab level uses them today.** Today’s lab levels are all single-output sources, so a `fromGateId`-only key would behave identically. But composite key `${fromGateId}:${fromPinIndex}` is the right shape because (a) input-node outputs and gate output pins are uniformly identified by the pair, and (b) any future 3-output gate (e.g. MUX-like in lab levels) gets per-pin accounting for free. **Lesson:** when adding a counter keyed on "source identity" in a system that *may* gain multi-output sources, key by the (sourceId, pinIndex) pair from day 1. Single-pin sources will collapse to `fromGateId:0` and the cost is zero; multi-output sources get correct accounting forever. Don’t inherit single-source assumption into the new code.

3. **Day 94’s `reasons.join(‘; ’)` composite-message format scales linearly to N constraint axes without touching the join code.** Adding the maxFanOut clause was a single `reasons.push(...)` after the existing `gateHardCap` and `mustIncludeGate` blocks. The L48 triple-violation rejection (`"4 gates exceeds hard cap of 3; fan-out 3 exceeds budget of 2."`) emerges automatically from the same join. Day 94 lesson #1 (preserve byte-exact message formats when rewriting a validator) paid off again — the Day 94 L44 7-NAND probe still passes byte-exact because the join produces identical output when `reasons.length === 1`. **Lesson:** when designing a validator that may grow more constraint axes, the multi-reason accumulator + ordered join pattern is the cheapest forward-compatible shape. New constraints land as new `reasons.push(...)` blocks; nothing else changes.

4. **The HUD "three static chip slots" pattern stays right at N=3, but N=4 will force a dynamic container.** Day 94 chose two static `#lab-constraint` / `#lab-constraint-2` spans over a dynamic flex container because N=2 didn’t justify the indirection. Today the same call extends to N=3 with one new `<span>` + one new render branch — still cheaper than a dynamic container. But the next constraint axis (whatever Cycle 6 Lab Bench IV adds) will make N=4, and 4 chips at standard mobile widths (375px) start to wrap awkwardly. **Lesson:** the static-slot pattern is correct at N ≤ 3 for known-shape HUDs. At N=4 or where order becomes data-driven, switch to a dynamic `<div>` container with `chips.forEach(append)`. Don’t pre-emptively dynamic-ize; do it when the next axis lands.

5. **A fresh BUILD-Day-3 harness can hit 42/42 on first run when the harness primitives are mature.** Day 107 had 3 self-bugs; Day 108 had 1; Day 109 had 0. Reason: the Day 107/108 harnesses already worked out the field names (`gs.gates`, `gs.nextId`, `startLevel(id)`, `gs.wireManager.wires`, `gs.wireManager.nextId`), the IO node access pattern (`gs.inputNodes`/`gs.outputNodes`), and the lab-state reset pattern (`gs._lab.attempts=0; gs._lab.exhausted=false`). Today’s harness was a recombination of Day 107’s build-identity + Day 108’s sw-fetch + Day 94’s composite-message assertion + Day 84’s level-shape probe. **Lesson:** harness primitives compound — don’t reinvent IO-node access or lab-state reset every BUILD day. Treat the prior day’s harness as a library of working idioms and recombine; new probes are the marginal cost, scaffolding is free.

6. **`maxFanOut` is the FIRST validator constraint that depends on wires rather than gates.** Day 84’s `gateHardCap` reads `this.gates`. Day 94’s `mustIncludeGate` reads `this.gates`. Day 109’s `maxFanOut` reads `this.wireManager.wires`. This means: a level with zero gates but many wires (e.g., passthrough from input to multiple outputs) can VIOLATE maxFanOut while satisfying every gate-counting constraint. L46 explicitly exploits this — the failure case has 0 gates and 3 wires. **Lesson:** when adding a constraint, identify whether it reads the gate population or the wire population. Wire-population constraints can trigger on "degenerate" circuits (zero gates, only wires), which means the validator must gracefully handle `this.gates.length === 0`. Today’s `placed = this.gates.filter(...)` doesn’t crash on empty arrays, but a future wire-aware constraint that accidentally assumes `gates.length ≥ 1` would.

7. **Cold-start level-card count grew silently from 45 to 50 today — that’s expected and is the canary for "new chapter added".** Day 102 lesson #6 flagged that the staircase test should explicitly probe `cards.length` per tier, not just the overflow count. P9.2 today asserts `cardCount === 50` (45 + 5). Without that explicit assertion, the +5 would have been silent slow-creep clutter. Future BUILD days that add levels should bump this number in the harness in lockstep. **Lesson:** when adding levels, update the cold-start card-count assertion in the SAME commit. Each BUILD day that ships new levels is a new canonical card count; failing to track it lets level-bloat sneak past audits.

## Day 94 — Cycle 4 BUILD Week, Day 3: Lab Bench II Composite Constraints

- **Preserve byte-equivalent message formats when you rewrite a validator.** Day 84's QA harness asserted on specific rejection strings (`Submission rejected: 5 gates exceeds hard cap of 4.`). When Day 94 promoted the validator from short-circuit to accumulate, the natural API change would have been to return a *reasons array* and force callers to format. That would have broken Day 84 + Day 92 regression assertions on dozens of lines. The right move: keep the public contract `{ok, message?}` exactly, switch internal structure to a `reasons[]` array, and join `reasons` with `; ` when forming the message. When there's exactly one reason, the join is a no-op and the message string is byte-identical to Day 84. Lesson: the validator's TYPE shape is more important than its INTERNAL shape when downstream tests grew around the type.
- **Two static chip slots beat a dynamic chip container for a known-small N.** The cleanest engineering instinct for “up to N chips” is a flex container with `appendChild` per chip. For N=2 today (and no plan for N=3 until Lab Bench III), that's over-engineering. Two static sibling `#lab-constraint` / `#lab-constraint-2` spans are: (a) zero CSS work because they share `.lab-chip.lab-constraint` styling, (b) zero DOM-shape change for Day 84 single-string levels (`#lab-constraint-2` just sits hidden), and (c) make the QA harness simpler (assert on two known element IDs, not a dynamic child count). Move to a dynamic container only when you actually need N≥3, not when N≥2 is *theoretically possible*.
- **`labConstraint: string | string[]` polymorphism is the cheapest backwards-compat shape.** Day 84 levels declare `labConstraint: '🧱 NAND only — universal gate practice'`. Day 94 needed two chips. Three viable shapes: (1) introduce `labConstraint2` as a sibling field, (2) replace `labConstraint` with `labConstraints: string[]` and migrate Day 84 levels, (3) make `labConstraint` polymorphic (`string | string[]`) and normalise at the render site. Picked #3 because: it's a one-line ternary at the consumer, Day 84 level data stays byte-identical (zero migration), and future levels can declare either shape. Lesson: when a single field needs to grow from cardinality 1 to cardinality N, polymorphic-with-render-time-normalisation beats both “new sibling field” and “migrate everything”.
- **The composite chip layout exposes a future Lab Bench III shape early.** With two chips visible at once, the lab HUD strip starts to feel like a “constraint manifest” — a list of rules the design must satisfy. Lab Bench III's multi-stage shape (two distinct submit windows with different rule sets) wants the same vocabulary: a constraint strip per stage, with stage tabs above. Today's two-chip strip is the visual proof-of-concept for that vocabulary. Don't paint over the chip-strip pattern in Day 95+ — it's the seed.
- **A NAND-only half-adder is the right “high-friction” composite level.** L44 has `availableGates: ['NAND']` (palette restriction) and `gateHardCap: 6`. The 5-NAND solution is famous but non-obvious to a player who hasn't seen it. The hint chain guides the player to the canonical NAND-only half-adder identity in 3 escalating steps. The hard cap of 6 leaves exactly 1 gate of headroom — the player can experiment with 1 wasted gate without rejection, but a 6→NAND solve also “works” cleanly. Lesson: a composite-constraint level should have headroom for a near-optimal solve, not require the optimal solve. Players prefer “tight but not impossible”.
- **For composite levels, prefer truth tables with non-obvious factoring.** L45's MUX is a classic; the *obvious* solve is `NOT(S) AND A` OR `S AND B` = 4 gates without XOR. `mustIncludeGate: ['XOR']` forces the player into a less-obvious shape: `A XOR ((A XOR B) AND S)` = 3 gates including 2 XORs. The constraint isn't just “must use XOR” — it's a *teaching moment* that XOR-based MUX is a real identity. Lesson: when a `mustIncludeGate` constraint maps onto a known mathematical identity, the level becomes pedagogically valuable, not just artificially restrictive.
- **`reasons.join('; ')` formatter scales naturally to N≥2 reasons.** Today only 2 constraint axes can fire simultaneously (`gateHardCap` and `mustIncludeGate`). Day 94's accumulator would equally handle 3, 4, N reasons because the format string is reasons-agnostic. When Lab Bench III adds two more constraint axes (e.g. `maxDepth`, `allowedGateTypes` distinct from `availableGates`), the message format requires zero changes — just push more reasons. Don't optimise for exactly-2; the joiner already handles arbitrary cardinality.

## Day 92 — Cycle 4 BUILD Week, Day 1: Module Split Phase 1 (gates.js as ES module)

- **The right Day-1-of-cycle move is the structural one that unblocks the rest of the week, not the most user-visible one.** Cycle 4's roadmap had 5 candidates; gates.js-as-ES-module was the only one that (a) had been deferred across two Prune reviews, (b) had a clean zero-fan-out leaf to extract, and (c) makes every subsequent module-split easier without touching gameplay. Shipping it Day 1 means Tuesday's Tournament Worker + Wednesday's Lab Bench composite constraints + Thursday's Onboarding readout UI + Friday's Snapshot Library can rest on the now-half-converted-but-stable boundary. Lesson: when a backlog item has been deferred twice, putting it at the front of the next BUILD week's roadmap is almost always the right call — the deferral cost compounds and Phase 2 only gets harder if Phase 1 keeps slipping.

- **`<script type="module">` + global rebind shim is the safest possible ES-module migration path for a vanilla multi-file codebase.** I considered three approaches: (1) convert gates.js to ESM and add `import` statements to all 8 consumers (touches every file, hard to revert), (2) wrap gates.js in a namespace object and migrate callers gradually (adds indirection without ESM benefits), (3) convert gates.js to ESM and install exports on `window` so classic-script consumers see them as globals (one-file diff + 12-LOC tail block + 1-tag change in index.html). Option 3 ships in a single commit, is byte-equivalent at runtime, and leaves the consumers untouched. The only constraint: ES-module exports can't be evaluated by a classic-script loader (SyntaxError), so the file MUST load as `<script type="module">`. Lesson: when the migration's blast radius is N consumers, prefer the approach that touches 0 of them — the global rebind shim is exactly that.

- **Module-script deferred-execution order is the entire compatibility argument.** Module scripts evaluate AFTER HTML parsing completes but BEFORE `DOMContentLoaded` fires. Classic scripts evaluate eagerly during parsing. So the sequence becomes: classic scripts (`wires.js` ... `main.js`) parse and execute their top-level code (class declarations register, but no method bodies run) → HTML parsing finishes → module `gates.js` evaluates and installs `window.Gate` etc → `DOMContentLoaded` fires → `main.js`'s init handler constructs `new Game()`, which invokes constructors that call `new Gate(...)` via the now-installed global. Pre-implementation I ran a grep audit confirming that none of the 8 classic-script consumers reference `Gate` / `GateTypes` / `IONode` / `roundRect` at the top level (only inside method bodies). That grep audit IS the safety proof — if any consumer had a top-level `class Foo extends Gate {}` or `const x = GateTypes.AND`, this approach would not work. Lesson: before converting a foundational module, audit consumers for top-level references; if there are none, deferred-module ordering covers you for free.

- **A `tools/module-health.js` ESM column is the single best regression guard for module-split progress.** Adding 18 LOC to the existing module-health script gives every future audit day a single-line answer to "how much of the codebase is ESM?" — stdout summary now reads `ESM=1/10`. If a future BUILD or PRUNE day accidentally reverts `gates.js` to classic syntax (e.g., a rewrite that drops the `export` keywords), the next module-health run drops to `ESM=0/10` and the diff in `specs/module-health.md` will show ✅ → — in the gates.js row. Lesson: structural progress without a tracking metric is invisible to future-me; the tracking metric should be a single number that's easy to grep.

- **Harness selectors and accessor paths drift across days even when the app code doesn't.** I burned three iterations on the Day 92 harness: (a) `gs.weeklyTournament.backend` instead of `gs.tournamentBackend` for the Day 83 adapter, (b) `gs.wires.length` instead of `gs.wireManager.wires.length` (Day 91 already noted this), (c) `gs.progress.completed['1']` instead of `gs.progress.levels['1']` (Day 91 used the right path). None of these reflect app changes — they're harness drift because my mental model conflates the public game-engine surface with the internal data model. Lesson: when writing a CDP harness, start by `cat`ing the most recent passing harness (`qa-reports/day-N-qa.cdp.js` for N = lastBuildDay) and lifting the accessor paths verbatim. Don't write the L1-solve flow from first principles every time.

- **Module conversion bumps both cache-bust AND SW version — don't bump just one.** The cache-bust query string (`?v=1780272000`) invalidates the HTTP cache for `index.html`'s consumers; the SW `CACHE_NAME` (`signal-circuit-v61`) invalidates the SW's precache for the next activation. If I'd bumped only one, the SW's offline-cached classic-script `gates.js` would still get served on the next install, even though the new `index.html` requests it as a module — and the browser would refuse to evaluate ES-module syntax as a classic script. Lesson: when changing the *load shape* of a file (classic → module, type attribute change), the SW version bump is non-negotiable. It's not a bonus; it's the only way the next SW activation evicts the old shape.

- **`tools/module-health.js` regex needs to allow `export ` prefix on declarations.** The Day 86 DECL_RE only matched `^(class|function|const|let|var)\s+...`. After adding `export` to gates.js, the scanner would have undercounted `gates.js`'s globals to 0 (because `^export class Gate` doesn't match `^class Gate`). I updated DECL_RE to `^(?:export\s+)?(class|function|const|let|var)\s+...` so it picks up both forms. Without this fix, the per-symbol fan-in/fan-out arithmetic for `gates.js` would be silently corrupt. Lesson: when adding ESM to a regex-based static analyzer, the FIRST regex to update is the declaration scanner; the second is the export-detection regex; both have to land in the same commit.

- **Module split Phase 1 = 1 file, not 2 or 3.** I was tempted to bundle `wires.js` (fan-out=2, fan-in=5) into the same day. That would have doubled the migration's blast radius (two `<script type="module">` tags + two `window.X = X` shims + double the regression surface) without doubling the structural payoff. Day 1 of a 5-day BUILD week should ship 1 cleanly-verified module, not 2 marginally-verified ones. Lesson: when in doubt about scope for the first conversion in a multi-step migration, ship the simplest possible version; the audit-trail value of "Day N did exactly X" beats the LOC efficiency of "Day N did X+Y but the QA report has to disambiguate which failure mode came from which".

## Day 65 — Prune Week 1, Day 3: Code Cleanup
- **Static dead-code detection on JS needs three passes, not one**: A first pass looks for `function name(` declarations whose name appears nowhere else. A second pass walks each `class` block and asks the same question of methods. The third pass is the trap: methods with names like `update`, `pan`, `setTimeout` will always look unused because those tokens occur as JS keywords or browser globals everywhere. - **Verification has to exercise the removed surface**: Headless puppeteer asserted `setMute`/`pan`/`worldToScreen`/`showDailyConfig`/`getById`/`getMasteryChallenge`/`distToSegment` are all `undefined` on `window.game` (or globally).
